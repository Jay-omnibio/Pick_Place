# Plot Guide: How To Read Run Plots And Debug Logs

This guide explains each plot generated by `tools/plot_run_metrics.py`, what each line means, what "good" behavior looks like, and how to read the main console debug lines.

## Fast Diagnostics (Text First)

For quick triage before looking at plots, run:

```bash
python tools/analyze_run_diagnostics.py --csv logs/run_YYYYMMDD_HHMMSS.csv --save-report --plot
```

What it gives:
- Phase transition timeline with retry/failure reasons.
- Pick-side/Place-side descend drift summary (`x/y/z` start->end and mins).
- Command-vs-motion sanity (`move_y` sign vs actual `ee_y` drift).
- Optional PNG: `logs/plots/run_.../diagnostics_descend.png`.

## Where To Find Outputs
- Run CSV: `logs/run_YYYYMMDD_HHMMSS.csv`
- Plot folder: `logs/plots/run_YYYYMMDD_HHMMSS/`
- Text diagnostics report: `logs/reports/run_YYYYMMDD_HHMMSS_diagnostics.md`

## Common Axis Meaning
- `Step` (x-axis): control loop iteration index.
- `Meters` (y-axis for position/error plots): distance or position values.
- `Command` (y-axis for action plot): commanded movement/gripper values.

## Console Debug Cheat Sheet

### `[HB]` (heartbeat)
Printed every N steps (from config/CLI). This is the main quick health line.

Typical fields:
- `phase=...`: current phase.
- `x=... y=... xy=.../thr z=.../thr`: current errors vs thresholds (phase-dependent).
- `step=a/b`: phase timer progress.
- `stall=...`: no-progress counter.
- `conf=... vfe=... gate=c:v`: confidence/VFE and phase-gate status.
- `bt=... bt_reason=...`: BT status and active reason.

Good:
- Error terms trend down in the current phase.
- `stall` stays low or resets after improvement.
- `gate=1:1` near phase transitions.

Warning:
- Errors flat/increasing for long windows.
- `stall` keeps growing.
- `gate` often `0` near transition points.

### `[CtrlDbg]` (controller-level)
Shows low-level command/IK behavior:
- `req_move` vs `applied_move`: requested vs actually applied Cartesian step.
- `dq_raw` vs `dq_applied`: IK raw joint delta vs clamped joint delta.
- `scales=pos/yaw/top/null`: per-phase gain scales.
- `gripper=READY|MOVING/open|close`

Good:
- `applied_move` close to `req_move` most of the time.
- `dq_applied` not constantly saturated.

Warning:
- Large persistent gap between requested/applied.
- `dq_raw >> dq_applied` for long windows (constraint/singularity pressure).

### `[Phase]` and `[AI-BT]`
- `[Phase] A->B ...`: actual transition event, with key errors at switch time.
- `[AI-BT] recover reason=...`: BT-triggered recovery branch activated.

Good:
- Phase order is logical and mostly forward.
- Recoveries are occasional, not continuous.

Warning:
- Repeated loops (`Reach -> Align -> Reach`) without progress.
- Frequent recoveries with same reason.

## 1) `01_reach_metrics.png` - Reach Metrics
Lines:
- `distance_to_object`: `||s_obj_rel||` (norm of object-relative belief vector).
- `reach_error`: `||s_obj_rel - [0, 0, -0.08]||` (error to pre-grasp relative target).

How to interpret:
- `reach_error` is the main one for current logic.
- Lower is better.

Good trend:
- `reach_error` decreases over time with limited oscillation.

Warning signs:
- Flat for long windows (stagnation).
- Big repeated spikes (recovery loops or unstable approach).

---

## 2) `02_object_relative_belief.png` - Object Relative Belief Components
Lines:
- `s_obj_rel_x`: believed object offset in x, relative to EE.
- `s_obj_rel_y`: believed object offset in y, relative to EE.
- `s_obj_rel_z`: believed object offset in z, relative to EE.

Target meaning:
- Current pre-grasp target is near `[0, 0, -0.08]`.
- So expected near end of Reach:
  - `s_obj_rel_x -> 0`
  - `s_obj_rel_y -> 0`
  - `s_obj_rel_z -> -0.08`

Good trend:
- `x,y` move toward 0.
- `z` moves toward `-0.08` (not necessarily 0).

---

## 3) `03_ee_belief.png` - EE Belief Components
Lines:
- `s_ee_x`, `s_ee_y`, `s_ee_z`: believed EE position in world frame.

How to interpret:
- This is belief-space EE trajectory, not direct ground truth.
- Use with plot 5 to compare belief vs true motion.

Good trend:
- Smooth, purposeful changes matching approach/lift phases.

Warning signs:
- Sharp sawtooth or repeated back-and-forth without improving reach metrics.

---

## 4) `04_actions.png` - Action Commands
Lines:
- `action_move_x`, `action_move_y`, `action_move_z`: commanded EE movement per step.
- `action_grip`: gripper command.
  - `1` = close
  - `0` = no-op
  - `-1` = open

How to interpret:
- Shows what policy/controller asks robot to do each step.
- Useful to diagnose oscillation or mode switching.

Good trend:
- Reach: mostly movement commands.
- Descend/CloseHold: `action_grip` goes to `1`, movement becomes small/targeted.
- Lift: positive `action_move_z` with `action_grip=1`.

Warning signs:
- Repeated fast sign flips in `action_move_z` (may indicate recovery fighting constraints).

---

## 5) `05_true_positions.png` - True World Positions
Lines:
- Solid lines:
  - `true_ee_x`, `true_ee_y`, `true_ee_z` (actual EE world position)
- Dashed lines:
  - `true_obj_x`, `true_obj_y`, `true_obj_z` (actual object world position)

How to interpret:
- This is ground truth from simulator.
- Best plot for checking if robot actually moved where expected.

Good trend:
- EE approaches object position (with expected pre-grasp offset in z), then lift/place behavior appears.

Warning signs:
- EE nearly flat while commands are non-zero (kinematic/posture stall).

---

## 6) `06_phase_contact_gripper.png` - Phase, Contact, Gripper Observation
Lines:
- `phase`: task phase index (ytick labels follow current AI pipeline such as Reach/Align/Descend/LiftTest/Transit/MoveToPlaceAbove/...).
- `obs_contact`: binary contact observation.
  - `1` means contact detected.
- `escape_active`: recovery state flag.
  - `0` = off
  - `1` = escape trigger
  - `2` = recovery active
- `obs_grip`: observed gripper width.

How to interpret:
- This is the timeline of state machine and contact events.

Good trend:
- Phase transitions happen in expected order.
- Contact rises near grasp attempts.
- `escape_active` appears occasionally, not continuously.

Warning signs:
- Stuck in Reach for long runs.
- Frequent Reach<->Align loops without reaching `CloseHold`.
- Continuous recovery activation.

---

## 7) `07_descend_diagnostics.png` - Descend Drift Diagnostics
Panels:
- Descend `x/y/z` errors (+ optional `true_descend_xy_error`).
- Commanded `action_move_x/y/z`.
- `true_ee_y` with `s_obj_rel_y/x` for drift visibility.

Good trend:
- Descend `x,y,z` all reduce in same phase window.
- `action_move_y` sign and `true_ee_y` drift direction should agree.

Warning signs:
- `y` error increasing while descend continues.
- `action_move_z` near zero for long intervals with no XY convergence.

---

## 8) `08_runtime_health.png` - Runtime Timing Health
Panels:
- `loop_dt_ms`, `loop_dt_ema_ms`, `loop_target_ms`.
- `obs_age_ms`, `obs_stale_warn`.
- `loop_overrun_count`.

Good trend:
- Loop dt near target, low jitter/overruns, low observation age.

Warning signs:
- Frequent overruns or stale warnings; this can look like control instability.

---

## 9) `09_ai_diagnostics.png` - AI Confidence/VFE/Gates
Panels:
- `ai_obs_confidence`
- `ai_vfe_total`
- `ai_phase_conf_ok`, `ai_phase_vfe_ok`, `ai_phase_gate_ok`
- Retry/failure event flags

Good trend:
- Confidence stable, VFE not continuously high, gate flags mostly `1`.

Warning signs:
- Gate flags oscillate or remain `0`, many retry/failure spikes.

---

## 10) `10_active_refs_phase_step.png` - Active References And Phase Step
Panels:
- Active reach/descend refs from runtime belief/config.
- `phase_step` counter.

Good trend:
- Refs are what you expect for the current task setup.
- `phase_step` resets cleanly at transitions.

---

## 11) `11_reach_path_xy.png` - Pick-Side XY Path (Start To Descend End)
Panels:
- Left: world XY path from first Reach start to end of first Descend
  - `true_ee_xy`, `obs_ee_xy`, optional `belief_ee_xy`
- Right: object-relative XY path during the same pick-side window
  - `obs_obj_rel_xy`, optional `belief_obj_rel_xy`, `reach_ref_rel_xy`, `desc_ref_rel_xy`

How to interpret:
- This is the best plot for your "sine-wave" question during approach+descend.
- It shows whether XY is converging cleanly all the way to descend completion.

Good trend:
- Smooth XY with limited side-to-side reversals through descend.
- Relative XY path converges toward `reach_ref_rel_xy` first, then remains controlled near `desc_ref_rel_xy`.

Warning signs:
- Repeated left-right loops in world XY.
- Relative XY path circles around refs without settling during descend.

---

## 12) `12_phase_split_metrics.png` - Phase-Split Metrics (Per Segment)
What it shows:
- One subplot per phase encountered in the run.
- For each phase, all contiguous segments are overlaid using local segment step (`0..N`) instead of global time.

Why it helps:
- Removes mixed-phase clutter.
- Lets you compare repeated attempts of the same phase directly (for example first `Reach` vs retry `Reach`).

Typical metrics per phase:
- Reach: `obs_reach_xy_error`, `obs_reach_z_error`
- Align/PreGraspHold/Descend/CloseHold/LiftTest: `obs_descend_xy_error`, `obs_descend_z_error`
- MoveToPlaceAbove: `obs_preplace_error`, `obs_preplace_xy_error`
- DescendToPlace/Open/Retreat/Done: `obs_place_error`, `obs_place_xy_error`

Good trend:
- Later segments of same phase converge faster or to lower final error.

Warning signs:
- Same phase segments plateau at similar high error.
- Retry segments look worse than the first segment.

---

## 13) `13_retry_windows.png` - Retry-Centered Windows
What it shows:
- Zoomed windows around retry start events (`ai_retry_reason` becomes non-empty).
- Three panels per retry:
1. Phase timeline around retry
2. Key error norms (`reach_xy`, `descend_xy`, `preplace_norm`, `place_norm`)
3. Contact/grip/Z context (`obs_contact`, `action_grip`, `true_obj_z`, `true_ee_z`)

Why it helps:
- Makes retry trigger cause obvious (slip, loss of contact, no progress, place regression).

Good trend:
- Retry is rare and error quickly decreases after retry start.

Warning signs:
- Retry occurs with contact loss while carrying.
- Error norms increase right before retry and do not recover after retry.

---

## Quick Debug Recipe
1. Check `01_reach_metrics.png` for overall convergence.
2. Check `06_phase_contact_gripper.png` for phase/contact timeline.
3. If stuck, inspect `04_actions.png` for oscillation.
4. Confirm true motion in `05_true_positions.png`.
5. Use `02_object_relative_belief.png` to see which axis is blocking progress.
6. Match plot behavior with `[HB]` and `[CtrlDbg]` lines at the same step windows.
7. Use `11_reach_path_xy.png` to verify if Reach motion is direct or wave-like.
8. Use `12_phase_split_metrics.png` to compare first attempt vs retry attempts phase-by-phase.
9. Use `13_retry_windows.png` to identify exact retry trigger cause.
